# Agent 脚手架与组件全景索引

本文件是 Agent 构建的**总览索引**，把 D1–D9 分散规范汇总为「脚手架 vs 运行时组件」清单，并说明**用途**与**什么时候该加**。

适用场景：
- 用户问「Agent 里有哪些模块 / 脚手架 / 组件」
- 从 0 到 1 规划 Agent 时做 v1 裁剪
- 认知辅导：用产品经理能懂的话解释工程概念
- Review 前对照「是否缺关键组件」

详细规范仍读对应 Domain 文件；本文件不重复展开实现细节。

---

## 概念区分

| 概念 | 通俗解释 | 类比 |
| --- | --- | --- |
| **脚手架（Scaffolding）** | 项目骨架：目录、配置入口、规则文档、部署脚本 | 盖楼前的图纸和施工规范 |
| **组件（Components）** | 运行时模块：loop、工具执行、权限、memory 等 | 楼里的水电、门禁、消防系统 |

---

## 一、脚手架清单

脚手架解决：**项目怎么组织、规则写在哪、怎么跑起来**。

### 1.1 项目目录与入口

| 脚手架 | 用途 | 什么时候加 | 详见 |
| --- | --- | --- | --- |
| Agent 主入口（CLI / API / Web） | 接收用户输入，启动一次对话 | 从 0 开始就要有 | D9 |
| 配置目录（如 `.claude/`、`config/`） | 集中放规则、技能、环境配置 | 第 1 天就建 | D3 |
| 工具注册表 / 工具目录 | 统一管理所有工具定义 | 有第 1 个工具时 | D2 |
| 日志与审计目录规范 | 规定什么必须记录、存哪 | 准备接真实用户前 | D9 |
| Docker / 部署脚本 | 一键启动、环境一致 | 要给别人用或上线时 | D9 |

### 1.2 规则与知识脚手架

| 脚手架 | 用途 | 什么时候加 | 详见 |
| --- | --- | --- | --- |
| CLAUDE.md（项目级） | 写「这个项目 Agent 该怎么工作」的全局规则 | 项目启动时 | D3 |
| 目录级 CLAUDE.md | 不同模块有不同规则 | 项目变大、模块职责清晰后 | D3 |
| `.claude/rules/` 模块化规则 | 把大文档拆成多个小文件 | 规则超过 ~200 行或多人维护时 | D3 |
| Skills（技能包） | 复杂任务标准流程（带工具限制、隔离上下文） | 某类任务重复出现 | D3、D6 |
| Commands（快捷命令） | 简单、一步到位的操作 | 高频小操作 | D3 |

**Skill vs Command 选择：**
- 简单快捷 → Command
- 复杂、要隔离上下文、要限制工具 → Skill（带 `allowed-tools`）

### 1.3 开发与上线脚手架

| 脚手架 | 用途 | 什么时候加 | 详见 |
| --- | --- | --- | --- |
| CI/CD 流水线 | 自动跑测试、安全扫描 | 2 人以上协作或要上线 | D3、D9 |
| Session 隔离规范 | 「写代码」和「Review 代码」用不同会话 | AI 生成 + Review 并存时 | D3 |
| 测试目录（单元 / 集成 / 工具 mock） | 验证 Agent 行为可重复 | 接真实业务前 | D9 |
| 环境变量 / 密钥管理模板 | API Key、MCP 密钥不写进代码 | 接任何外部服务时 | D2、D8 |

## 二、运行时组件清单

组件是 Agent **真正跑起来**时需要的模块。按「最小可用 → 生产级」分层。

### 2.1 第一层：最小 Agent（Demo / PoC）

| 组件 | 用途 | 什么时候加 | 详见 |
| --- | --- | --- | --- |
| Agentic Loop | 模型反复「想 → 调工具 → 看结果 → 再决定」 | 必须有 | D1 |
| stop_reason 控制 | 用 API 结构化信号决定继续或结束 | 第一天就要 | D1 |
| System Prompt 构建器 | 组装系统指令、角色、边界 | 有 Agent 就要有 | D4 |
| Message 历史管理 | 保存对话、工具调用、工具结果 | 有 Agent 就要有 | D1 |
| Tool Schema 注册 | 告诉模型有哪些工具、参数格式 | 有工具就要有 | D2 |
| Tool Executor | 真正调用 API、查库、读写文件 | 有工具就要有 | D2、D6 |
| Tool Result 回填 | 把工具结果正确塞回消息历史 | 有工具就要有 | D1、D2 |

### 2.2 第二层：可靠 Agent（内测 / 小流量）

| 组件 | 用途 | 什么时候加 | 详见 |
| --- | --- | --- | --- |
| Iteration Budget | 限制最多循环几次，防止死循环 | 工具调用 > 1 次时 | D6 |
| Tool Timeout | 单个工具不能无限等 | 接网络/数据库时 | D6 |
| 结构化错误返回 | 区分「查不到数据」vs「系统挂了」 | 接真实数据源时 | D2 |
| Retry / Failover | 网络抖动、限流时自动恢复 | 接外部 API 时 | D6 |
| Context Budget | 控制塞进模型的 token 总量 | 对话变长或文档变多时 | D5、D7 |
| Case Facts Block | 关键信息结构化保存，不被摘要冲掉 | 客服、审批、不能丢细节的场景 | D5 |
| Scratchpad | 中间结果写文件/DB，不靠反复摘要 | 长任务、多步骤任务 | D1、D5 |

### 2.3 第三层：多 Agent / 复杂编排

| 组件 | 用途 | 什么时候加 | 详见 |
| --- | --- | --- | --- |
| Coordinator | 拆任务、派活、汇总结果 | 单 Agent 工具 > 5 个，或跨领域 | D1 |
| Subagent / Task 委派 | 专职 Agent 做专项 | 任务可并行、上下文需隔离 | D1 |
| 最小上下文传递 | 子 Agent 只拿任务相关片段 | 有多 Agent 就必须有 | D1 |
| 结构化上报 | 子 Agent 失败返回错误类别 | 有多 Agent 就必须有 | D1、D5 |
| Session / Fork / Resume | 继续对话或分叉实验 | 长项目、探索性任务 | D1、D3 |
| Prompt Chaining | 步骤固定 A→B→C | 流程稳定、输入输出可预期 | D1 |
| Dynamic Decomposition | 根据中间结果决定下一步 | 任务复杂度未知 | D1 |

### 2.4 第四层：工具与安全（接权限 / 写操作）

| 组件 | 用途 | 什么时候加 | 详见 |
| --- | --- | --- | --- |
| Tool Policy | 每轮决定哪些工具可用/隐藏/全禁 | 用户说「只分析别执行」 | D7 |
| Plan Mode 只读白名单 | 默认禁止写，只允许读工具 | 规划、Review、只读分析 | D3、D7 |
| Hooks | 工具执行前后用代码拦截关键规则 | 金额、权限等关键业务规则 | D1 |
| Tool Guardrails | 检测同一工具同一参数反复失败 | 工具多、易重复调用 | D6 |
| Admin / Non-admin 权限 | 普通用户不能发邮件、接 MCP 等 | 多用户产品 | D7、D8 |
| 文件路径 Allowlist / Denylist | 模型给的路径不能直接读写 | 任何文件读写 | D7、D8 |
| Workspace Containment | 限制 Agent 只在当前项目目录内操作 | 本地/IDE 类 Agent | D7 |
| Prompt Injection Hardening | 外部内容用边界标记包起来 | 接网页、邮件、文档 | D7、D8 |
| Secret Redaction | 日志和输出里不泄露 API Key | 接密钥时 | D8 |

### 2.5 第五层：Memory / Context 引擎

| 组件 | 用途 | 什么时候加 | 详见 |
| --- | --- | --- | --- |
| Context Engine | 统一管理压缩、预算、切换模型后重算 | 长对话、多文档 | D6 |
| Compression Preflight | 压缩前先注入不能丢的内容 | 有 memory 或 Case Facts 时 | D6 |
| Memory Provider | 长期记忆的可插拔后端 | 需要跨会话记住用户偏好 | D6 |
| Memory Manager 生命周期 | session 切换、压缩、委派时同步 memory | 有 memory 就要有 | D6 |
| Provenance（信息溯源） | 记忆/检索带来源、时间、置信度 | 需要可追溯、可审计 | D5 |
| RAG / Search 服务 | 从知识库检索再回答 | 领域知识超出模型训练数据 | D5、D7 |

### 2.6 第六层：MCP 与扩展

| 组件 | 用途 | 什么时候加 | 详见 |
| --- | --- | --- | --- |
| MCP Manager | 连接、管理多个 MCP Server | 要接第三方工具生态 | D2、D7 |
| MCP Schema 预算 | 限制渲染到 prompt 的 schema 大小 | MCP 工具很多时 | D2、D7 |
| MCP Bridge | 把内部工具暴露给外部 runtime | 多 runtime 共享工具面 | D6 |
| readOnlyHint / destructiveHint | 标注工具是否只读/破坏性 | 每个 MCP 工具定义时 | D2、D7 |

**MCP**（Model Context Protocol）：标准协议，让 Agent 像插 USB 一样接外部工具服务。

### 2.7 第七层：Skill 与学习（Hermes 风格）

| 组件 | 用途 | 什么时候加 | 详见 |
| --- | --- | --- | --- |
| Skill 索引与统计 | 知道哪些 skill 常用、可组合 | skill 数量变多 | D6 |
| Related Skills 关联 | 推荐相关 skill | 技能图谱复杂时 | D6 |
| Skill 文本不可信处理 | 外部 skill 也当不可信输入 | 允许用户/第三方上传 skill | D6、D8 |
| Message Sanitization | 修复非法 tool call、孤儿 tool result | 生产 runtime | D6 |
| Post-turn Hooks | 每轮结束后的持久化、审计 | 需要完整 turn 级追踪 | D6 |

### 2.8 第八层：产品化 / 上线（Odysseus 风格）

| 组件 | 用途 | 什么时候加 | 详见 |
| --- | --- | --- | --- |
| FastAPI / 后端路由 | HTTP API、用户认证 | Web 产品 | D7、D9 |
| 前端 UI | 聊天、文件、任务管理 | 面向非技术用户 | D7 |
| 审计日志 | 谁、何时、调了什么工具 | 上线前必须 | D9 |
| 观测指标 | 延迟、失败率、token 消耗 | 上线前 | D9 |
| Human Escalation | 超权限、政策空白、用户要求转人工 | 客服/审批类业务 | D1、D5 |
| 回滚方案 | 出问题能退回上一版本 | 正式上线 | D9 |
| 运维文档 / 交接清单 | 新人能接手 | 团队 > 1 人或要交接 | D9 |

## 三、按阶段裁剪：v1 该做什么

| 阶段 | 最少需要什么 | 典型触发条件 |
| --- | --- | --- |
| **PoC** | Loop、1–3 个工具、System Prompt | 「先跑通一个 demo」 |
| **内测** | 预算、超时、结构化错误、Case Facts | 接真实数据库/API |
| **小流量** | Tool Policy、Plan Mode、Hooks、路径安全 | 能写文件、发邮件、花钱 |
| **生产** | 权限矩阵、审计、观测、回滚、测试 | 多用户、要追责、要 SLA |

阶段推进路径：

```
PoC → 内测 → 小流量 → 生产
  │      │       │        │
  Loop   预算    权限     Memory
  工具   错误    Hook     MCP
  Prompt 分类   注入防护  审计/观测
```

## 四、产品经理必记五条红线

1. **关键规则不能只写 prompt** → 要 Hook / 权限 / schema 在代码里强制执行（D1、D8）
2. **单 Agent 工具 ≤ 5 个** → 多了拆多 Agent（D1、D2）
3. **「没查到」和「系统挂了」必须分开** → 否则 Agent 会瞎编（D2）
4. **Plan Mode 要真禁写工具** → 不能只在 prompt 里说「别写文件」（D3、D7）
5. **网页/邮件/记忆/skill 都是不可信输入** → 要有边界包装（D7、D8）

完整反模式见 `references/anti-patterns.md`。

## 六、使用本索引时的输出建议

### 架构规划模式

按用户场景输出：
1. 当前阶段（PoC / 内测 / 小流量 / 生产）
2. v1 必做脚手架 + 组件（从本索引裁剪）
3. v2 再做的演进项
4. 每项对应的业务风险与验收方式

### 认知辅导模式

按「一句话结论 → 业务比喻 → 工程拆解 → 常见误区 → 可执行下一步」输出；优先引用本索引中的表格项，再深入读对应 Domain。

### Review 模式

对照本索引检查项目是否缺：
- 第二层：结构化错误、预算（接真实数据时）
- 第四层：权限、路径、注入防护（有写操作时）
- 第八层：审计、观测、回滚（准备上线时）
